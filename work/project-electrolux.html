The same metric, read by ten people, produced ten different numbers. The problem was not the data: it was the absence of a shared language for displaying it.

The Situation

Electrolux was a global company with distributed teams, each managing their own reporting in isolation. Excel files, Google Sheets, Power BI dashboards built without a visual standard: every team had a version of the truth, but no team had the same version as any other.

Leadership could not consolidate a view across regions. Analysts spent time reformatting instead of analyzing. Reports that should have enabled decisions became sources of confusion and reconciliation work.

Dojo was brought in to build dashboards from a centralized Data Lake. The brief was to connect the data and make it readable. I saw an opportunity to do something more permanent: build a system, not just screens.

The Mandate

Create Power BI dashboards connected to the Data Lake that gave Electrolux teams across LATAM a unified way to read their operational and strategic data. Deliver across 5 project waves, each covering a different business area.

Beyond the dashboards themselves, I identified a gap the brief had not named: if every dashboard was custom-built from scratch, the same fragmentation would return the moment Dojo left. Teams would diverge again.

So I built a template system and a documentation guide alongside the dashboards. The goal was a standard any team could replicate without a designer in the room.

The System

Three contexts, one language.

Different roles have fundamentally different relationships with data. An executive needs immediate orientation. An operations manager needs granular tracking. An analyst needs to interrogate. I designed a distinct visual template for each context, all sharing the same tokens, typography, and color conventions.

01 /

Executive Dashboard

Designed for leadership and board-level visibility. Maximizes signal, minimizes noise. One screen should answer "how are we doing?" without any interaction. KPIs are displayed as large, unambiguous values with trend indicators and goal comparisons. Drill-down is available but never required to understand the summary.

Operational Dashboard

Designed for day-to-day management and team leads who monitor output and identify anomalies quickly. Higher data density than the executive view, with time-series comparisons and performance breakdowns by category, team, or period. Built for repeated daily use, not occasional review.

Analytical Dashboard

Designed for data analysts and controllers who need to interrogate the data, not just read it. Supports filtering, cross-referencing, and variance analysis. The most complex context visually, but structured so that complexity serves exploration rather than creating confusion for non-analyst users.

Process

Understanding before designing.

Phase 01 · Understand

Business Understanding sessions with stakeholders

Every dashboard started with a Business Understanding meeting with the relevant stakeholder. The goal was to capture the business rules behind the data before touching the design: what does this metric actually measure, who is the primary reader, what decision does this dashboard enable, and what does "wrong" look like when data is misread?

These sessions prevented the most common failure mode in data visualization: building something technically correct that nobody uses because it answers the wrong question.

Stakeholder Interviews Business Rules Mapping

Phase 01 · Align

Alignment with data analysts before design

Before creating any visual, I worked with Dojo and Electrolux data analysts to understand the shape of the data: what was available in the Data Lake, what transformations were needed, what dependencies existed between metrics, and where the data had known gaps or inconsistencies.

Design decisions made without this alignment fail in implementation. I preferred to understand constraints upfront and design within them rather than discover them during the binding phase.

Data Analysis Constraint Mapping Cross-team Alignment

Phase 02 · Build

Dashboard design and data binding

With business rules and data structure mapped, I designed the dashboards in Power BI following the template system. Each dashboard went through a binding process with the data team to connect visual components to live Data Lake queries.

Template-based design compressed build time: the visual decisions for color, typography, spacing, and chart type were already made. Each dashboard required only the structural decisions specific to its content.

Power BI Data Binding Template System

Phase 03 · Validate

Validation with teams before handoff

Each completed dashboard was reviewed with the team that would use it: checking whether the visual representation matched their mental model, whether labels and terminology reflected how they actually talked about the metrics, and whether the default view answered what they needed without requiring configuration.

Small naming and framing adjustments at this stage prevented large confusion after delivery. Validation was not optional: it was part of the process for every wave.

The System

Designed to outlast the project.

The biggest risk in a consulting engagement is that standards disappear when the consultants leave. I built the system to survive without me.

Deliverable

Reusable Dashboard Templates

Instead of building each dashboard from scratch, I developed a template library covering the three contexts: Executive, Operational, and Analytical. Each template defined the visual structure, chart selection criteria, color usage rules, and typography. Any new dashboard started from the appropriate template, which enforced consistency without requiring designer involvement in every new request.

Deliverable

Usage Documentation Guide

I created a written manual for the visualization system: how to apply the templates, when to use each chart type, color conventions for status and comparison, naming standards for metrics and labels, and decision rules for choosing between dashboard contexts. The goal was a document detailed enough that a Power BI user with no design background could build a new dashboard and have it look like it belonged to the same system.

Parallel Project

Nidec Dashboard System

Simultaneously with the Electrolux engagement, I developed a parallel data visualization system for Nidec, applying the same methodology: Business Understanding sessions, analyst alignment, template-based design, and a dedicated documentation guide. The two projects shared a process and a design philosophy but were implemented independently to match each company's data structure, terminology, and organizational context.

Scale

From LATAM to everywhere.

Waves

Five deployment cycles across LATAM

The project ran in 5 waves, each covering a different business area or regional scope within Electrolux LATAM. Each wave followed the same process: Business Understanding, analyst alignment, design and binding, team validation. The template system made each wave faster than the one before.

Adoption

Standard presented to the global board

After completing the LATAM deployment, the visualization standard was reviewed at board level and selected as the reference model for Electrolux globally. The templates and documentation guide moved from a regional project to a worldwide standard, applied across all Electrolux companies.

Design Decision

System design over one-off execution

The reason the standard could scale globally was that it was built as a system from the start, not as a collection of individual dashboards. Visual consistency came from rules and templates, not from having a designer present in every room. That was the intended outcome.

Outcomes

Regional project, global impact.

5 Project waves delivered across LATAM with no rework on the visual standard
Global Visualization standard adopted worldwide across all Electrolux companies after board review
0 Designers required to replicate the system after the documentation guide was delivered

A dashboard that requires a designer to explain it has not finished being designed. The goal was a system where the documentation was the designer, and the templates were the consistency. By the time the LATAM project ended, a designer was not needed for the next one to look the same. That is the best possible outcome for a visual system built inside a consulting engagement.
